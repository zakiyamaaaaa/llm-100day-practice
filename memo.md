自律型エージェント（ReAct）
例えば、ユーザーから「明日の朝、皇居を走りたいんだけど、雨が降りそうなら代わりに代々木公園のコース情報を教えて」と頼まれたとします。
このとき、エージェントは以下のように自律的に思考と実行のループを繰り返す必要があります。1. 【思考】 まず明日の天気を調べる必要があるな。➔ get_weather(location="皇居") を実行しよう。2. 【実行結果】 「明日の朝の皇居周辺は、降水確率80%の雨」と返ってくる。3. 【思考】 雨か。それならユーザーの指示通り、代わりに代々木公園の情報を取得しよう。➔ get_running_course_info(location="代々木公園") を実行しよう。4. 【実行結果】 代々木公園のコース情報が返ってくる。5. 【思考】 必要な情報はすべて揃った。ユーザーに最終回答を構築しよう。
このように、「思考（Reason） ➔ 行動（Act） ➔ 観察（Observation）」 のサイクルを、LLMが「これで解決した」と判断するまで** while ループ等で自律的にグルグル回し続ける設計パターン**を ReAct（Reasoning and Acting） と呼びます。
→agent_react_loop.pyで実装。

1. 状態（State）の維持と永続化の複雑化
   今回実装したループは、メモリ上の messages 配列に履歴をアペンドしていくだけのシンプルなものでした。しかし、実際のアプリでは：
   • ユーザーが途中で「あ、やっぱり今のナシで」と**会話を巻き戻した（タイムトラベル）**とき、状態をどう復元するか？
   • スマホアプリのタスクキルやバックグラウンド移行、あるいはサーバーの再起動を挟んでも、**会話の途中の状態（コンテキスト）から正確に再開（永続化）**できるか？
   これを手書きの配列管理だけで実装しようとすると、コードが急速にスパゲッティ化します。
2. 人間の承認（Human-in-the-Loop）の挟み込み
   エージェントが「社外PC持出許可（様式第4号）」の関数を自律的に呼び出すコードを書きましたが、もしこれが「社内サーバーの再起動」や「経費の自動送金」といった重大なアクションを伴うツールだった場合、勝手に実行されたら大問題になります。
   • ツールを実行する直前で**処理を一時停止（ブレークポイント）**し、管理者がSlackや管理画面から「承認」ボタンを押したら処理を再開する。
   この「一時停止と再開」のステート管理を手書きの for や while ループの中で美しく制御するのは至難の業です。
3. 複雑な循環グラフ（マルチエージェント）への拡張
   「天気を調べる担当エージェント」と「ドキュメントを探すRAG担当エージェント」が、お互いに成果物を受け渡しながら合議制で回答を決める、といったマルチエージェント駆動をしようとすると、手書きのループでは制御フローの記述が限界を迎えます。
   🕸️ 解決策：グラフ構造による状態管理（LangGraphなど）
   これらの課題をスマートに解決するために、現在のLLM開発シーン（2026年現在）では、エージェントのフローを 「グラフ（Node = 処理、Edge = 遷移線）」 として定義し、中央の 「State（状態オブジェクト）」 を全員で書き換えながら自律駆動させるフレームワークが主流となっています。その代表格が LangGraph です。
   私たちが今回書いた ReAct ループも、モダンなフレームワークの設計思想に当てはめると、裏側では以下のような「有限状態機械（ステートマシン）」として厳密にモデリングされています。

ベクトル検索も、万能ではない。次のような場合は、ベクトル検索ではうまくいかない。
ケース1：型番・エラーコード・固有の記号
ユーザーが 「ERR*404の発生原因を教えて」 と質問したとします。
• 人間の期待: 「ERR_404: ページが見つかりません」というドキュメントを引っ張ってきてほしい。
• ベクトル検索の挙動: 「ERR_404」という文字列全体の「意味」を解釈しようとします。結果、「ERR_500」や「SYS*エラー」といった、「なんとなくシステムエラーについて書いてある別のドキュメント」のほうがベクトル空間上で近くに配置され、誤ってヒットしてしまうことが多々あります。LLMにとって、404 も 500 もただの数字の羅列でしかなく、文字としての「完全一致」を重視しないためです。
ケース2：固有名詞・特定の書式（様式第○号など）
ユーザーが 「様式第4号の申請先は？」 と聞いたとします。
• ベクトル検索の挙動: 「様式第1号」「様式第2号」もすべて「申請書のフォーマット」という酷似した意味を持つため、高確率で混同します。ピンポイントで「4号」を狙い撃ちするのが大の苦手なのです。

🤝 解決策：ハイブリッド検索（Hybrid Search）
この弱点を克服するために、現在のモダンなRAGアーキテクチャでは、**「ベクトル検索（意味）」と「伝統的なキーワード検索（文字の完全一致）」を同時に実行し、双方の検索結果をブレンドして上位を並び替える「ハイブリッド検索」**が必須のデファクトスタンダードとなっています。

ユーザーの質問 ──┬─> [ 1. ベクトル検索 (Chroma DB) ] ──> スコア（距離）
│
└─> [ 2. キーワード検索 (BM25等) ] ───> スコア（出現頻度）

                └───> [ 3. 二つの結果を統合 (RRF) ] ───> LLMへのコンテキスト

→これをやってみたのが、hybrid_search.pyとhybrid_test.py。
後者がベクトル検索のみ、前者がターゲットを絞って、それに合致したらボーナスポイントをつける。
これは、BM25のような伝統的なキーワード検索のアルゴリズムを使って、ベクトル検索の結果にスコアを加算することで、より精度の高い検索結果を得ることができます。

ベクトル検索のスコア（コサイン距離：0.0〜1.0）と、BM25のスコア（出現頻度やレア度：0.0〜数十）は、単位もスケールも全く異なるため、単純に足し算することはできません。
この「リンゴとミカンを足し算できない問題」を、数学的な美しさで解決したのが RRF です。
📐 RRF (Reciprocal Rank Fusion) の仕組み
RRFの思想は驚くほどシンプルです。
「スコア（生数値）のことは一切忘れよう。それぞれの検索エンジンが出した『順位（ランキング）』の順位点だけで勝負させよう」 というアプローチです。
各ドキュメントのRRFスコアは、以下の数式で計算されます。
‭$$\text{RRF Score} = \frac{1}{k + \text{ベクトル検索の順位}} + \frac{1}{k + \text{BM25検索の順位}}$$‬‭‬‭‬
※ ‭$k$‬ は定数（一般的には 60 が世界標準）で、上位の順位の価値を高くしつつ、下位の順位のノイズでスコアが乱高下するのを防ぐ平滑化の役割を持ちます。
💡 なぜこの数式が天才的なのか？
• ベクトルの世界で 1位 ➔ ‭$\frac{1}{60 + 1} = 0.01639$‬‭‬
• BM25の世界で 1位 ➔ ‭$\frac{1}{60 + 1} = 0.01639$‬‭‬
