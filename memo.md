ベクトル検索も、万能ではない。次のような場合は、ベクトル検索ではうまくいかない。
ケース1：型番・エラーコード・固有の記号
ユーザーが 「ERR*404の発生原因を教えて」 と質問したとします。
• 人間の期待: 「ERR_404: ページが見つかりません」というドキュメントを引っ張ってきてほしい。
• ベクトル検索の挙動: 「ERR_404」という文字列全体の「意味」を解釈しようとします。結果、「ERR_500」や「SYS*エラー」といった、「なんとなくシステムエラーについて書いてある別のドキュメント」のほうがベクトル空間上で近くに配置され、誤ってヒットしてしまうことが多々あります。LLMにとって、404 も 500 もただの数字の羅列でしかなく、文字としての「完全一致」を重視しないためです。
ケース2：固有名詞・特定の書式（様式第○号など）
ユーザーが 「様式第4号の申請先は？」 と聞いたとします。
• ベクトル検索の挙動: 「様式第1号」「様式第2号」もすべて「申請書のフォーマット」という酷似した意味を持つため、高確率で混同します。ピンポイントで「4号」を狙い撃ちするのが大の苦手なのです。

🤝 解決策：ハイブリッド検索（Hybrid Search）
この弱点を克服するために、現在のモダンなRAGアーキテクチャでは、**「ベクトル検索（意味）」と「伝統的なキーワード検索（文字の完全一致）」を同時に実行し、双方の検索結果をブレンドして上位を並び替える「ハイブリッド検索」**が必須のデファクトスタンダードとなっています。

ユーザーの質問 ──┬─> [ 1. ベクトル検索 (Chroma DB) ] ──> スコア（距離）
│
└─> [ 2. キーワード検索 (BM25等) ] ───> スコア（出現頻度）

                └───> [ 3. 二つの結果を統合 (RRF) ] ───> LLMへのコンテキスト

→これをやってみたのが、hybrid_search.pyとhybrid_test.py。
後者がベクトル検索のみ、前者がターゲットを絞って、それに合致したらボーナスポイントをつける。
これは、BM25のような伝統的なキーワード検索のアルゴリズムを使って、ベクトル検索の結果にスコアを加算することで、より精度の高い検索結果を得ることができます。

ベクトル検索のスコア（コサイン距離：0.0〜1.0）と、BM25のスコア（出現頻度やレア度：0.0〜数十）は、単位もスケールも全く異なるため、単純に足し算することはできません。
この「リンゴとミカンを足し算できない問題」を、数学的な美しさで解決したのが RRF です。
📐 RRF (Reciprocal Rank Fusion) の仕組み
RRFの思想は驚くほどシンプルです。
「スコア（生数値）のことは一切忘れよう。それぞれの検索エンジンが出した『順位（ランキング）』の順位点だけで勝負させよう」 というアプローチです。
各ドキュメントのRRFスコアは、以下の数式で計算されます。
‭$$\text{RRF Score} = \frac{1}{k + \text{ベクトル検索の順位}} + \frac{1}{k + \text{BM25検索の順位}}$$‬‭‬‭‬
※ ‭$k$‬ は定数（一般的には 60 が世界標準）で、上位の順位の価値を高くしつつ、下位の順位のノイズでスコアが乱高下するのを防ぐ平滑化の役割を持ちます。
💡 なぜこの数式が天才的なのか？
• ベクトルの世界で 1位 ➔ ‭$\frac{1}{60 + 1} = 0.01639$‬‭‬
• BM25の世界で 1位 ➔ ‭$\frac{1}{60 + 1} = 0.01639$‬‭‬
